# ClaudeMonitor - Документ требований к продукту

## Видение продукта
Компактное desktop-приложение (Windows) для мониторинга использования Claude AI в реальном времени. Плавающее полупрозрачное окно поверх всех окон показывает процент использования по моделям, обратный отсчёт до сброса лимитов, статус инцидентов на стороне Claude, а также управление несколькими аккаунтами.

## Целевая аудитория
Активные пользователи Claude (claude.ai, Claude Code) с несколькими аккаунтами, которым важно видеть текущий расход лимитов по каждому аккаунту и оперативно переключаться между ними.

## Основные функции

### 1. Мониторинг использования (реализовано)
- Авторизация через CookieBridge (основной поток) или Playwright (fallback)
- Отображение процента использования по категориям: Сессия, Все модели, Opus, Sonnet, Cowork, OAuth Apps
- Обратный отсчёт до сброса лимитов (обновление каждую секунду)
- Автообновление каждые 3 минуты
- Компактный режим (только процент сессии + время активного аккаунта)
- Иконка в трее с динамическим процентом активного аккаунта
- Сохранение позиции окна между перезапусками

### 2. Мониторинг инцидентов Claude Status (реализовано)
- Опрос Atlassian Statuspage API (`status.claude.com/api/v2/incidents/unresolved.json`) каждые 2 минуты
- Кликабельные лейблы активных инцидентов внизу окна (цвет по severity)
- Попап с деталями инцидента по клику на лейбл
- Кастомные toast-уведомления при появлении/завершении инцидентов

### 3. Мультиаккаунтность (реализовано)
- Поддержка до 4 аккаунтов claude.ai
- Хранение куков каждого аккаунта в `accounts/<id>.json`
- Таблица аккаунтов в развёрнутом окне: email/имя, % сессии (5ч), % недельного лимита (7д), кнопка переключения →
- Активный аккаунт отмечен зелёной точкой
- Горячее переключение — без перезапуска
- Фоновый поллинг всех аккаунтов каждые 5 минут
- Автоопределение email/имени аккаунта через API после логина
- Кнопка "+ Добавить" в секции аккаунтов
- Переключение через контекстное меню (правая кнопка)

### 4. CookieBridge интеграция (реализовано)
- Локальный HTTP-сервер на порту 19224 (запускается вместе с приложением)
- Эндпоинт `POST /site-cookies/claude` — получает куки от расширения Chrome
- Эндпоинт `GET /health` — статус для popup расширения
- При получении куков: автоматически определяет аккаунт по `sessionKey`, сохраняет, устанавливает активным если нет других
- Расширение CookieBridge (BOZEX-Photo/extension) расширено поддержкой домена `claude.ai` с `serverOverride: "http://localhost:19224"`

### 5. Интеграция с TrayConsole (реализовано)
- Управление через Named Pipes: show/hide/refresh/relogin/shutdown
- Heartbeat для мониторинга процесса

### 6. Поддержка HTTP/HTTPS прокси (реализовано)
- Файл `proxy.json` рядом с приложением: `{"url": "http://127.0.0.1:10808"}`
- Fallback: переменные окружения `HTTPS_PROXY` / `HTTP_PROXY`
- Прокси применяется ко всем исходящим запросам: curl_cffi subprocess (fetch/identity/ping/login) и urllib (Statuspage)
- Пункт контекстного меню "Прокси: &lt;url&gt;" открывает диалог для смены URL (пустое значение — выключить прокси)
- Нужен, когда claude.ai/status.claude.com недоступны напрямую и доступ идёт через локальный прокси

## Технический стек
- Python 3 + PyQt5
- curl_cffi (HTTP для API claude.ai)
- urllib.request (HTTP для Statuspage API — без доп. зависимостей)
- http.server (CookieBridge сервер на 19224 — без доп. зависимостей)
- Playwright (fallback логин через Chrome)
- PyInstaller (сборка в .exe)
- psutil (управление процессами)

## Технические ограничения
- Только Windows (pywin32, Named Pipes)
- Один файл приложения: `usage_monitor.py`
- Без новых зависимостей
- Statuspage API публичный, без аутентификации
- Chrome хранит куки в едином хранилище профиля — два аккаунта одновременно возможны только в разных профилях Chrome

## Структура хранения аккаунтов
```
accounts/
  acc_<md5>.json     # Playwright storage_state с куками
accounts_meta.json   # { active: "acc_...", accounts: [{id, email, name, added_at}] }
```
